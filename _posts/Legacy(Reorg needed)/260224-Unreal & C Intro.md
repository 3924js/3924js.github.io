오늘은 보다 다른 내용으로 더 들어갔다.

* 언리얼 친해지기
	* 게임 개발자 직군
		* 게임 기획자
			* 어떤 게임을 만들지 구상하고 그에 맞게 계획을 수립하는 사람
			* 스토리라인, 레벨 디자인, 게임 규칙, 재미와 BM 등을 구성하고 구체화
		* 아티스트(그래픽 디자이너, 애니메이터)
			* 눈으로 보이는 것들을 실제로 만드는 사람들
			* 컨셉 아티스트: 컨셉화, 원화를 제작해 디자인의 방향성을 제시
			* 3D 모델러: 게임에 사용될 캐릭터, 몬스터, 물체등 3D모델을 만드는 사람
			* 애니메이터: 모델에 움직임을 넣는 사람
			* TA(Technical Artist): 그래픽 작업의 기술적인 부분을 담당, 아티스트와 프로그래머의 중간다리 역할.
			* 아트 디렉터(Art Directer)
		* 프로그래머
			* 기획에 맞게 규칙과 기능들을 구현
			* 클라이언트 프로그래머
				* 실제 플레이어에게 보이는 세계를 구현
				* 실제 레벨 로직, 이벤트, UI등 기능의 구현
				* 대개 최적화는 클라이언트 프로그래머의 직무
			* 서버 프로그래머
				* 온라인 게임에서의 일관성과 공정성을 보장하며 서버 기능을 구현
				* 치트나 해킹등 부정행위에 대한 방어
				* 서버의 안정성을 위해 야간근무, 대기가 필요할 수도 있음. 게임을 흥하게는 못해도 망하게는 할 수 있음.
		* 사운드 아티스트
			* 게임에 사용될 음향 효과, 배경음악등 사운드를 담당
		* PD(Producer)
			* 게임 개발의 총괄, 스튜디오의 리더. (인디팀, 중소규모 기업의 경우 PD가 CEO일 경우가 많음)
			* PM(Project manager)들과 함께 프로젝트 개발 기간동안 꾸준한 진행상황 체크와 확인
			* 일정 관리, 예산 조율, 대외적인 면담이나 미팅등 개발 외의 업무까지 담당.
		* 조직 구조가 잘 잡혀진 큰 기업의 경우 각 팀 간의 역할이 뚜렷하나, 소규모 인디 팀의 경우 경계가 느슨할 수 있음.
* 언리얼 엔진 에디터 구성
	* 뷰포트(Viewport): 게임 월드를 시각적으로 보여주고 조작할 수 있게 해줌
		* 마우스 왼쪽을 누른채로
			* 마우스 회전: 방향
			* wasd: 이동
			* qe: 상하이동
	* 아웃라이너(Outliner): 월드에 존재하는 오브젝트들의 목록
	* 디테일(Detail): 선택된 오브젝트의 상세사항을 보여줌(위치, 설정, 재질 등등)
	* 컨텐츠 브라우저(Content Browser): 프로젝트에서 쓸 수 있는 에셋들을 보여주는 탐색기
	* 메뉴 바(Menu Bar): 별도의 메뉴들 집합(프로젝트 저장, 빌드, 창, 개인 설정 등등)
* 종류
	* 월드(World): 모든 액터와 구성 요소들을 포함하는 최상위 컨테이너로, 게임을 관리하는 역할
	* 레벨(Level): 액터나 물체가 배치될 수 있는 공간으로, 월드는 여러 레벨로 구성될 수 있음.
	* 액터(Actor): 언리얼 엔진에서 오브젝트를 지칭하는 말, 사물, 카메라, 조명등 많은 것들이 이 분류에 들어감.
		* 폰(Pawn): 플레이어 혹은 AI를 통해 조종 가능한 액터, 캐릭터나 적등의 구현에 사용 가능
			* 캐릭터(Character): 플레이어 컨트롤러가 장착되어 플레이어가 조종 가능한 폰.
* 좌표계
	* XYZ 3개의 축을 통해 위치를 표현
	* World와 Local 두개의 좌표계가 있음
		* World: 게임 월드에 대한 절대적인 좌표계
		* Local: 게임 오브젝트의 중심을 기준으로 한 상대적인 좌표계
		* 하나의 좌표도 World 좌표, 그리고 각각 다른 위치에 있는 오브젝트에 따라 Local 좌표는 다 다름. 필요에 따라 좌표계를 선택해야함
	* 기즈모(Gizmo)를 이용해 오브젝트를 바꾸는데도 사용할 수 있음
		* 위치(Position)
		* 회전(Rotation)
		* 스케일(Scale)
	* 머티리얼(Material)
		* 물체가 렌더링되어 보여지는 색감, 질감, 투명도 등을 조절
* C언어
	* 1-9 변수(Variable)
		* 변할 수 있는 수. (<-> 상수(Constant): 변할 수 없는 수)
			* unsingned: 부호 없이 0부터 양수만 표현. 음수가 없기 때문에 음수 표현에 쓰던 데이터 공간까지 양수 표현에 쓰면서 표현 가능한 양수 범위가 2배로 늘어남.
			* signed: 음수까지 포함하는 일반적인 자료형.
		* 오버플로우(Overflow)
			* 표현가능한 범위를 넘어가 가능한 영역 반대에 있는 값으로 넘어가는 것
				* ex) -128 <-> 127 까지 표현 가능하면 127+1 = -128
				* 반대로 아래로 내려가는건 언더플로우(Underflow)
					* -128 - 1 = 127
	* 1-10 ASCII(American Standarded Code for Information Interchange)
		* 컴퓨터에서 문자를 표현하기위해 각 문자에 일대일 대응되게 만들어놓은 인코딩 규약
			* ex) 'a' = 97, 'A' = 65, 'Z' = 90
			* 인코딩(Encoding): A에서 B로 변환 (문자 -> 숫자)
				* <-> 디코딩:(Decoding): B에서 A로 변환 (숫자 ->문자)
		* ASCII, EUC-KR등 여러 규격으로 언어 별로 파편화 되어있었으나, Unicode로 통일되어 현대에는 보통 Unicode 사용 (UTF-8이 사실상 표준, UTF-16, UTF-32도 있음.)
	* 1-11 입력(Input)
		* scanf(): 키보드로부터 입력을 받는 함수, printf()의 반대
			* ex) scanf("%d", &Num), scanf("%c", &MyChar);
	* 2-1 연산자(Operator)
		* 연산을 시키는 기호
			* 피연산자(Operand): 연산이 되는 기호
				* 1 + 2 -> 1,2 : 피연산자 / + : 연산자
		* 산술 연산자(Calculation operator)
			* +(덧셈), -(뺄셈), \*(곱셈), /(나눗셈), %(모듈로, 나머지, 정수에만 사용)
			* +와 -로만 이루어진 식에선 왼쪽에서 오른쪽 순서로 진행
				* \*, % 가 들어가면 연산 우선순위 가짐
				* 엄밀히 말하면 괄호나 브라켓도 우선순위가 높은 연산자
			* 복합대입연산자(Assignment operator)
				* 산술 연산자 + 대입연산자 (ex) num = num + 1  -> num += 1)
				* +=, -=, \*=, /=
		* 값(Value)
			* 리터럴, 존재하는 결과 그 자체(ex) 1, "hello", 3.14)
		* 문(Statement)
			* 값으로 이어지는 것들(ex) 1+2, 1.2-0.3)
		* 식(expression)
			* 컴퓨터가 동작할 수 있는 명렁어(ex) int num = 8;, while(true))
	* 2-2 증감,논리,비교 연산자
    	* 증감연산자(Increment and decrement operators): 하나의 항에서 증가 또는 감소를 일으키는 연산자
        	* ++, --
        	* 전치 연산자: ++Num -> Num이 호출되기 전에 증가 후 호출
        	* 후치 연산자: Num++ -> Num이 호출 된 이후 증가.
      	* 논리연산자(Logical Operators)
        	*  And, Or, Not등에 해당하는 논리연산을 수행하는 연산자
              	*  &&(and, 논리합)
              	*  ||(or, 논리곱)
              	*  !(not, 부정)
              	*  short-circuit: A && B에서 A가 false면, 혹은 A || B에서 A가 true라면, B는 확인하지 않고 넘어감 (성능상 유리)
           	*  논리 연산에 있어 0, null, \0 정도를 제외한 값이 있는 모든것(1,2 ,-1 ,3.14, 'a' 등)은 true로 연산
     	*  관계 연산자(relational operator)
        	*  대소비교등 두 항의 관계에 대해 작동하는 연산자
              	*  \>(greater)
              	*  \<(less)
              	*  <=(less or equal)
              	*  \>=(greater or equal)
              	*  ==(equal)
              	*  !=(not equal)


전날과 마찬가지로 기초적인 내용을 돌아볼 수 있는 시간이였다.

또한 현재 작성중인 TIL 페이지를 잘 정리하는 방법에 대해 여러 방식을 고려중이다.
1. 날짜별 정리
   * 매일 새로 배우는 내용을 각 날짜에 맞춰 파일로 업로드(현재 하는 방식)
   * 그날마다 새로운 파일로 업로드 하기에 편하게 투고 가능
   * 정보의 성격이나 토픽에 따라 분류되지 않았기에 나중에 정보를 검색하고 찾는데 불편할 수 있음
2. 범주별 정리
   * 각 정보를 그 주제에 맞게 묶어서 폴더 별로 정리
   * 여러 영역을 다루게 됬을 때 작성, 수정해야하는 파일의 갯수가 늘어남
   * 후일에 정보를 검색하고 접근하기에 보다 용이
현재는 여러 방식에 대해 고민중이라 1번 방식을 하고있으나, 빠른 시일 내에 2번 형식을 중심으로 재편해야 할 듯 하다. 장기적인 관리와 유지보수에는 2번 방식이 더 유리할 것이다.

또한 Github를 사용한 TIL 작성에 있어 블로그 형식을 이용하면 보다 나은 시인성과 접근성을 가질 듯 하다.
* Github pages를 이용해 블로그, 개인용 사이트 형식으로 페이지를 만들 수 있다.
  * 장점
    * 무료 호스팅: 여느 블로그 서비스들과 마찬가지로 일단 호스팅 자체는 무료이다.
    * 템플릿 사용, 확장성: Github에 올라와있는 많은 템플릿들을 활용할 수 있으며, 레포지토리에 기능들을 포함하는 식으로 확장해나갈 수 있다.
    * Git 활용: Github 페이지에 들어가 온라인으로 작성하는 것 뿐 아니라, 컴퓨터에서 commit을 통해 업로드할 수도 있다.
    * 코드 연결: Github에 올라와있는 코드, 레포지토리의 내용을 보여주는등
  * 단점
    * 유지보수 필요: 기본적으로 블로그 형식을 위해 만들어진 것은 아니며, 블로그 페이지에 쓸 레포지토리를 중심으로 필요한 기능들을 추가, 통합하는 것이기 때문에 글만 쓰고 수정하면 되는 네이버 블로그, velog같은 느낌이라 할 순 없다. 기능에 대한 이해와 노력을 필요로 한다.
    * 글쓰기의 불편함: Markdown을 바탕으로 작성하는 것 까진 보편적이고 쉬울 수 있으나, 사진과 동영상의 첨부, 파일의 관리가 그렇게까진 직관적이지 않을 수 있다. 당장에 소개된 방법들만 보더라도 이미지는 레포지토리나 별도의 드라이브에 올려 그 이미지의 링크를 첨부하는 형식으로 진행해야 한다.
TIL 레포지토리를 구축하는 초기라면 여러가지 시도를 해볼 수도 있겠으나, 장기적으로 가져가려면 하나의 정해진 틍을 만들어 가는 것이 좋을것이다.
